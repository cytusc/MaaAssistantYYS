## 1. 项目定位

MaaAssistantYYS 是一个面向《阴阳师》的自动化助手项目，目标是在 MAA 的基础框架能力之上，实现阴阳师的日常任务、活动任务、战斗流程和资源识别自动化。

项目不直接照搬 OAS 的 Python 任务体系，而是将 OAS 中可复用的阴阳师业务经验、资源定义和流程算法迁移到 C++/MAA 风格的低耦合架构中。

## 2. 核心目标

1. 复用 MAA 的设备控制、截图、图像识别和跨平台基础能力。
2. 删除或隔离明日方舟相关业务逻辑，建立阴阳师专用任务层。
3. 区分日常模块和活动模块，使活动更新时只改活动模块。
4. 使用面向对象和接口抽象降低耦合，避免单文件过大、职责混乱。
5. 支持 dry-run、单元测试和设备联调，保证每个小单元可验证。

日常模块的原则是稳定、少变、可配置，不应包含某期活动的临时判断。

### 3.2 活动模块

活动模块用于承载周期性变化的活动内容，例如：

- 爬塔活动
- 超鬼王
- 花车巡游
- 对弈竞猜
- 答题类活动
- 新版本限时副本

活动模块必须独立于日常模块，每期活动应尽量通过新增活动配置和活动类实现，不修改核心流程和日常任务。

**当前状态**：`YYSBaseActivity` 和 `YYSActivityManager` 骨架已实现，但无具体活动任务。活动样板资源（`I_ACTIVITY_ENTER`、`I_CHALLENGE`）已存在。

## 4. 非目标范围

当前阶段不优先处理：

1. 完整 GUI 产品化。
2. 所有阴阳师活动的一次性全量覆盖。
3. 复杂 AI 决策，例如自动配阵、自动御魂搭配。
4. 云端账号同步或多人协作服务。
5. 绕过游戏风控或破坏游戏协议的行为。

## 5. 验收标准

### 5.1 基础验收

- [x] 项目可以 CMake 配置成功。
- [x] `MaaYYS` 静态库可以构建成功。
- [x] 至少一个 dry-run 可执行程序可以跑通完整任务链（`OrochiDryRun`）。
- [x] 资源加载器可正常解析 JSON 配置（`TestResourceLoader`）。

### 5.2 架构验收

- [x] 日常模块和活动模块目录隔离。
- [x] 业务任务不直接依赖具体控制器或具体识别实现，只依赖抽象接口。
- [x] 单个任务文件应保持可读，过大时拆分为 Navigator、Selector、Handler 等小类。
- [x] Flow 流程组件被业务代码实际使用（御魂和探索任务均使用 Flow 体系）。
- [x] YYSConfig 配置类已定义并可使用（Sprint 2 完成，支持 JSON 加载/任务启停/优先级）。

### 5.3 功能验收

- [x] 御魂任务能够通过 dry-run 验证：进入、选层、开战、领奖。
- [ ] 御魂任务能在真实模拟器上完成至少一次完整流程（待 OpenCV 集成和设备联调）。
- [x] 至少一个其他日常任务（探索或悬赏封印）通过 dry-run 验证。

## 6. 关键约束

1. 所有任务都必须支持日志输出和失败返回。
2. 所有活动逻辑必须可单独禁用。
3. 活动模块不得修改日常模块内部逻辑。
4. 资源配置必须与代码解耦。
5. 每个新增模块必须有最小验证方式。

## 7. 已知限制

1. **MaaCore 未桥接**：`MaaControllerActionExecutor` 使用 `std::function` 回调，未绑定 MaaCore Controller；`MaaTemplateResolver` 的 `match_with_maa()` 返回 `nullopt`。
2. **MaaUtils 链接被注释**：CMakeLists.txt 中被注释，待 MaaCore 编译环境搭建后启用。
3. **占位 PNG**：除御魂外的模板 PNG 为 1x1 占位文件，需从游戏截图裁剪替换。
4. **真实 OCR 执行器**：`IOcrResolver` 接口已定义，但无真实实现。
